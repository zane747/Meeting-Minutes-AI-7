# Feature Specification: 使用者登入登出系統

**Feature Branch**: `006-user-auth`
**Created**: 2026-03-27
**Status**: Draft
**Input**: User description: "在 Meeting Minutes AI 專案中加入使用者登入登出系統"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 使用者註冊帳號 (Priority: P1)

使用者第一次使用系統時，需要註冊一個帳號。使用者提供帳號（使用者名稱）、密碼，系統建立帳號後自動登入並導向首頁。

**Why this priority**: 沒有帳號就無法登入，這是整個認證系統的基礎。

**Independent Test**: 可以透過開啟註冊頁面、填寫表單、送出後確認是否成功建立帳號並跳轉至首頁來獨立測試。

**Acceptance Scenarios**:

1. **Given** 使用者尚未擁有帳號, **When** 使用者填寫帳號與密碼並送出註冊表單, **Then** 系統建立帳號、自動登入、導向首頁
2. **Given** 使用者填寫的帳號已被他人使用, **When** 送出註冊表單, **Then** 系統顯示「此帳號已被使用」錯誤訊息，不建立帳號
3. **Given** 使用者未填寫必填欄位（帳號或密碼）, **When** 送出註冊表單, **Then** 系統顯示對應的欄位驗證錯誤

---

### User Story 2 - 使用者登入 (Priority: P1)

已註冊的使用者輸入帳號和密碼進行登入。登入成功後導向首頁，失敗則顯示錯誤訊息。

**Why this priority**: 登入是使用系統所有功能的入口，與註冊同為最高優先級。

**Independent Test**: 可以透過開啟登入頁面、輸入正確/錯誤的帳密、確認登入結果來獨立測試。

**Acceptance Scenarios**:

1. **Given** 使用者擁有已註冊的帳號, **When** 輸入正確的帳號與密碼並送出, **Then** 系統登入成功、導向首頁，頁面顯示使用者名稱
2. **Given** 使用者輸入錯誤的密碼, **When** 送出登入表單, **Then** 系統顯示「帳號或密碼錯誤」訊息，不揭露是帳號還是密碼有誤
3. **Given** 使用者輸入不存在的帳號, **When** 送出登入表單, **Then** 系統顯示「帳號或密碼錯誤」訊息（與上述相同，避免帳號枚舉）

---

### User Story 3 - 使用者登出 (Priority: P1)

已登入的使用者可以點擊登出按鈕，結束目前的登入狀態，並導向登入頁面。

**Why this priority**: 登出是認證系統不可或缺的一環，也是考核課題的明確要求。

**Independent Test**: 可以透過登入後點擊登出按鈕、確認是否回到登入頁面、再嘗試訪問受保護頁面被拒來獨立測試。

**Acceptance Scenarios**:

1. **Given** 使用者已登入, **When** 點擊登出按鈕, **Then** 系統清除登入狀態、導向登入頁面
2. **Given** 使用者已登出, **When** 嘗試訪問首頁或任何受保護頁面, **Then** 系統自動導向登入頁面

---

### User Story 4 - 未登入使用者被導向登入頁 (Priority: P2)

未登入的使用者嘗試訪問系統的任何功能頁面時，自動被導向登入頁面。登入成功後，導回原本想訪問的頁面。

**Why this priority**: 這是保護系統功能的基本機制，但優先順序略低於核心登入登出流程。

**Independent Test**: 可以在未登入狀態下直接訪問 `/meetings` 等頁面，確認是否被導向登入頁，登入後是否回到原頁面。

**Acceptance Scenarios**:

1. **Given** 使用者未登入, **When** 訪問 `/`（首頁）, **Then** 系統導向 `/login` 登入頁面
2. **Given** 使用者未登入且嘗試訪問 `/meetings/abc`, **When** 在登入頁成功登入, **Then** 系統導回 `/meetings/abc`
3. **Given** 使用者未登入, **When** 呼叫 API（如 `/api/meetings/upload`）, **Then** 系統回傳 401 Unauthorized

---

### Edge Cases

- 使用者名稱包含特殊字元或空白時，系統如何處理？（限定為英數字與底線）
- 密碼長度過短（少於 8 字元）或過長時，系統如何回應？（顯示驗證錯誤）
- 使用者在多個瀏覽器同時登入時，各 session 是否獨立？（是，各自獨立）
- 使用者的 session 過期後（長時間未操作），系統如何處理？（顯示「登入已過期，請重新登入」提示訊息後導向登入頁面）
- 同一帳號短時間內多次登入失敗時，系統是否有保護機制？（v1 暫不實作帳號鎖定，列為未來改進項目）
- 使用者在操作途中（如上傳音檔）session 過期時，如何處理？（API 回傳 401，前端應提示使用者重新登入）
- 瀏覽器禁用 Cookie 時，系統如何處理？（v1 不特別處理，登入會失敗但不會顯示 Cookie 相關提示）

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系統必須提供使用者註冊功能，包含帳號（使用者名稱）與密碼欄位
- **FR-002**: 系統必須驗證帳號唯一性，不允許重複帳號
- **FR-003**: 系統必須使用 bcrypt 演算法雜湊密碼後儲存，不可明文儲存
- **FR-004**: 系統必須提供登入頁面，使用者輸入帳號密碼後進行身份驗證
- **FR-005**: 系統必須在登入成功後建立 session，維持使用者的登入狀態
- **FR-006**: 系統必須提供登出功能，清除使用者的 session
- **FR-007**: 系統必須保護所有頁面與 API，未登入的使用者無法存取
- **FR-008**: 系統必須在未登入使用者訪問受保護頁面時，自動導向登入頁面
- **FR-009**: 系統必須在登入成功後，導回使用者原本想訪問的頁面
- **FR-010**: 帳號格式限定為 3~30 字元的英數字與底線
- **FR-011**: 密碼長度至少 8 字元
- **FR-012**: 當使用者的 session 過期時，系統必須顯示「登入已過期，請重新登入」提示訊息，並導向登入頁面
- **FR-013**: 已登入使用者訪問登入頁或註冊頁時，系統必須自動導向首頁（避免重複登入）
- **FR-014**: 系統必須在受保護頁面的回應中設定 `Cache-Control: no-store`，防止登出後透過瀏覽器「上一頁」存取快取內容

### Key Entities

- **User（使用者）**: 代表一個系統使用者，包含帳號（唯一，同時作為顯示名稱）、密碼雜湊值、建立時間等資訊
- **Session（登入狀態）**: 代表一次登入的有效期間，與特定使用者關聯，有過期時間

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 使用者可以在 1 分鐘內完成註冊並開始使用系統
- **SC-002**: 使用者可以在 30 秒內完成登入
- **SC-003**: 未登入使用者訪問任何受保護頁面時，100% 被導向登入頁面
- **SC-004**: 登出後，使用者無法透過瀏覽器「上一頁」存取受保護內容
- **SC-005**: 密碼在儲存時 100% 經過雜湊處理，資料庫中不存在明文密碼

### Security Requirements

- **SEC-001**: 密碼必須使用 bcrypt 演算法雜湊（自動 salt），資料庫中不得存在明文密碼
- **SEC-002**: Session Cookie 必須設定 `HttpOnly`（防止 JavaScript 讀取）、`SameSite=Lax`（基本 CSRF 防護）
- **SEC-003**: 登入失敗的錯誤訊息必須統一為「帳號或密碼錯誤」，不得區分帳號不存在或密碼錯誤（防止帳號枚舉攻擊）
- **SEC-004**: 登出必須使用 POST 方法（而非 GET），防止 CSRF 攻擊強制登出
- **SEC-005**: v1 使用 HTTP 部署（內部工具），未來正式環境應升級為 HTTPS 並將 Cookie 設定 `Secure` 屬性

### Known Limitations（已知限制，v1 不處理）

- 無帳號鎖定機制（多次登入失敗不會暫時封鎖帳號）
- 無密碼複雜度要求（僅檢查長度 ≥ 8，不要求包含大小寫/數字/特殊字元）
- Session 存在伺服器記憶體中，伺服器重啟後所有使用者需重新登入
- Session 有效期為固定 24 小時（非閒置計時），無論使用者是否持續操作都會在 24 小時後過期
- 不支援多節點部署（Session 不跨伺服器共享）
- 無密碼修改、密碼重設功能

## Clarifications

### Session 2026-03-27

- Q: 是否需要預設管理員帳號，還是開放自助註冊？ → A: 開放註冊，任何能連到系統的人都可自行註冊
- Q: 登入後顯示的名稱是帳號本身還是額外的顯示名稱欄位？ → A: 直接使用帳號（username）作為顯示名稱，不額外新增欄位
- Q: Session 過期時使用者正在操作，系統如何處理？ → A: 顯示「登入已過期，請重新登入」提示訊息後導向登入頁

## Assumptions

- 註冊頁面公開，任何能連到系統的人都可自行註冊帳號
- 本系統為單機部署的內部工具，不需要支援 OAuth、SSO 等外部認證機制
- v1 僅需基本的帳號密碼認證，不需要 email 驗證或忘記密碼功能
- 使用 server-side session（搭配 cookie）作為認證機制，符合現有 Jinja2 模板的架構
- 不需要角色或權限分級（v1 所有使用者權限相同），權限管理將作為獨立的後續功能
- Session 預設有效期為 24 小時
- 系統目前無需支援密碼修改功能（列為未來改進項目）
- 會議紀錄目前不綁定使用者（所有登入使用者可見所有會議），使用者與會議的關聯將作為後續功能
