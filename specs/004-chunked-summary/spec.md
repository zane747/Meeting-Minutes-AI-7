# Feature Specification: 分段摘要合併

**Feature Branch**: `004-chunked-summary`
**Created**: 2026-03-26
**Status**: Draft
**Input**: User description: "方案B — 將超長逐字稿分段送入 LLM，各段分別產生局部摘要後再合併為完整會議紀錄"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 長會議完整摘要 (Priority: P1)

使用者上傳一段超過 30 分鐘的會議錄音，系統完成語音辨識後，逐字稿長度超出 LLM 的 context window 限制。系統自動將逐字稿分成多個段落，分別產生局部摘要，最後合併為一份涵蓋所有議題的完整會議摘要與待辦事項。使用者在前端看到的摘要包含了會議中討論的每一個議題，不會遺漏任何段落。

**Why this priority**: 這是本功能的核心價值 — 解決目前長會議摘要只涵蓋前段內容、後段議題被截斷遺漏的問題。

**Independent Test**: 可透過上傳一段 40 分鐘的多議題會議錄音來測試，驗證產出的摘要是否涵蓋所有議題。

**Acceptance Scenarios**:

1. **Given** 一份 40 分鐘的會議錄音包含 5 個不同議題, **When** 系統完成轉錄與摘要, **Then** 最終摘要涵蓋全部 5 個議題的討論要點與決議
2. **Given** 逐字稿長度超過 LLM context window 限制, **When** 系統進行摘要, **Then** 系統自動啟動分段模式而非截斷逐字稿
3. **Given** 分段摘要完成, **When** 系統合併各段摘要, **Then** 合併後的摘要沒有重複內容且段落之間邏輯連貫

---

### User Story 2 - 短會議直接摘要 (Priority: P2)

使用者上傳一段短會議（10 分鐘以內），逐字稿長度未超過 LLM 的 context window。系統沿用現有的一次性摘要流程，不進行分段，避免不必要的多次呼叫。

**Why this priority**: 確保新功能不會影響短會議的既有體驗與效能，維持向下相容。

**Independent Test**: 上傳一段 5 分鐘的會議錄音，確認系統僅呼叫 LLM 一次即完成摘要。

**Acceptance Scenarios**:

1. **Given** 一份逐字稿長度在 context window 限制內, **When** 系統進行摘要, **Then** 系統採用單次摘要模式，行為與改版前一致
2. **Given** 短會議的逐字稿, **When** 摘要完成, **Then** 產出格式與長會議分段合併後的格式一致

---

### User Story 3 - 待辦事項完整提取 (Priority: P2)

使用者的會議中，不同時段分別提到了不同的待辦事項。系統在分段摘要後，能從各段分別提取待辦事項，合併後去重，產出完整的待辦清單。

**Why this priority**: 待辦事項是會議紀錄的重要產出之一，若遺漏會直接影響後續工作追蹤。

**Independent Test**: 上傳一段會議錄音，其中前半段與後半段各提到不同的待辦事項，驗證最終待辦清單包含所有項目。

**Acceptance Scenarios**:

1. **Given** 會議前半段提到 3 個待辦事項、後半段提到 2 個待辦事項, **When** 分段摘要合併完成, **Then** 最終待辦清單包含全部 5 個待辦事項
2. **Given** 不同段落提到相同的待辦事項, **When** 合併待辦清單, **Then** 重複項目被合併為一筆

---

### User Story 4 - 主題時段分析完整 (Priority: P3)

系統的語義分析（semantic_analysis）中的主題列表，能正確反映整場會議的所有主題及其對應時段，而非只有前段的主題。

**Why this priority**: 主題時段分析讓使用者能快速定位會議中感興趣的段落，完整性很重要但不如摘要本身關鍵。

**Independent Test**: 上傳多議題會議，檢查主題列表是否涵蓋所有時段的議題。

**Acceptance Scenarios**:

1. **Given** 一場涵蓋 6 個議題的會議, **When** 語義分析完成, **Then** 主題列表包含全部 6 個議題及其正確的時間區間

---

### Edge Cases

- 逐字稿剛好在 context window 邊界（如差 100 字就超過），系統如何判斷是否需要分段？
- 分段切割點恰好落在一個議題的中間，如何避免該議題的摘要被割裂或遺漏？
- LLM 在某一段摘要呼叫中失敗或超時，系統重試一次後仍失敗則跳過該段，以已完成段落產出部分摘要，並在摘要中標註遺漏的時段範圍
- 極短的逐字稿（如只有幾句話）被分到最後一段，如何避免產出無意義的摘要？
- 會議只有一個議題但非常冗長（如持續重複討論同一件事），分段合併後是否會產出重複的摘要？

## Clarifications

### Session 2026-03-26

- Q: 分段失敗處理策略？ → A: 重試一次後仍失敗則跳過該段，以已完成段落產出部分摘要，標註遺漏時段
- Q: 合併摘要的方式？ → A: 全部透過一次 LLM 呼叫合併（摘要整合 + 待辦去重 + 主題合併一起處理）
- Q: 最大支援會議長度？ → A: 最長 2 小時（約 8-12 段）

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系統 MUST 在逐字稿長度超過 LLM context window 可用容量時，自動切換至分段摘要模式
- **FR-002**: 系統 MUST 將逐字稿按語義邊界（如時間段落或靜默間隔）分割成多個段落，每段長度不超過 LLM context window 可用容量
- **FR-003**: 系統 MUST 對每個分段分別呼叫 LLM 產生局部摘要、局部待辦事項與局部主題分析
- **FR-004**: 系統 MUST 在所有分段摘要完成後，透過一次 LLM 呼叫將各段的局部摘要、待辦事項與主題列表一起整合為完整的會議摘要
- **FR-005**: 系統 MUST 在合併呼叫中由 LLM 語義去重待辦事項，產出完整待辦清單
- **FR-006**: 系統 MUST 在合併呼叫中由 LLM 整合各段的主題時段分析，產出涵蓋整場會議的主題列表
- **FR-007**: 系統 MUST 在逐字稿長度未超過限制時，沿用現有的單次摘要流程
- **FR-008**: 系統 MUST 在分段摘要過程中，向前端回報進度（如「正在處理第 2/4 段」）
- **FR-009**: 系統 MUST 確保合併後的摘要輸出格式與單次摘要的格式完全一致
- **FR-010**: 系統 MUST 在分段切割時保留相鄰段落的重疊內容（上下文重疊），以避免議題被切割導致語意不完整
- **FR-011**: 系統 MUST 在單一分段的 LLM 呼叫失敗時，自動重試一次；重試仍失敗則跳過該段，以其餘段落產出部分摘要
- **FR-012**: 系統 MUST 在產出部分摘要時，於摘要中明確標註遺漏的時段範圍，讓使用者知道哪些段落未被摘要

### Key Entities

- **逐字稿段落（Transcript Chunk）**: 從完整逐字稿分割出的一個片段，包含文字內容、起始時間、結束時間、段落序號
- **局部摘要（Chunk Summary）**: 單一段落經 LLM 處理後產出的摘要、待辦事項與主題分析
- **合併摘要（Merged Summary）**: 所有局部摘要整合後的最終會議摘要，格式與現有單次摘要一致

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 40 分鐘以上的會議摘要涵蓋率達到 100%（所有討論議題均出現在摘要中）
- **SC-002**: 長會議的待辦事項提取完整率達到 95% 以上（與人工校對比較）
- **SC-003**: 分段摘要的總處理時間不超過單次摘要時間的 N+1 倍（N 為段落數）
- **SC-004**: 短會議（context window 內）的處理時間與改版前無顯著差異
- **SC-005**: 合併後的摘要可讀性評分不低於單次摘要（無明顯重複或割裂）
- **SC-006**: 系統能完整處理最長 2 小時的會議錄音並產出涵蓋所有議題的摘要

## Assumptions

- 現有的 LLM（Ollama + gemma2）支援多次獨立呼叫且回應格式穩定
- 逐字稿中包含時間戳（如 `[MM:SS - MM:SS]`），可作為語義分段的依據
- 分段重疊的上下文不需要超過單段容量的 10%
- 本功能不改變前端的顯示方式，僅改變後端的摘要生成邏輯
- 本功能不影響 Gemini 遠端模式，僅作用於 Ollama 本地摘要流程
- 系統設計支援最長 2 小時的會議（約 8-12 個分段），超過此長度不保證處理品質
