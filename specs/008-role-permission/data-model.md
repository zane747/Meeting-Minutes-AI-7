# Data Model: 權限管理

---

## 修改的實體

### User — 新增 `role` 欄位

| 欄位 | 型別 | 限制 | 說明 |
|------|------|------|------|
| `role` | Integer | NOT NULL, 預設 3 | 角色等級：1=超級管理員, 2=管理員, 3=一般使用者 |

**角色等級對照**:

| 值 | 名稱 | 權限 |
|----|------|------|
| 1 | 超級管理員 | 全部功能 + 管理中心 + 角色變更 |
| 2 | 管理員 | 管理等級 3 帳號 + 查看下級會議 + 未來可發公告 |
| 3 | 一般使用者 | 自己的會議 + 修改自己的密碼 |

**等級規則**：數字越小權限越高。`role <= N` 表示「等級 N 以上都可以」。

### Meeting — 新增 `created_by` + `visibility` 欄位

| 欄位 | 型別 | 限制 | 說明 |
|------|------|------|------|
| `created_by` | String(36) | FK → User.id, nullable | 上傳者 ID（既有資料為 null） |
| `visibility` | String(20) | NOT NULL, 預設 "private" | 可見性：public / private / same_level |

**可見性規則**:

```
查詢會議列表時的過濾邏輯：

等級 1（超級管理員）：
  → 看全部（不過濾）

等級 2（管理員）：
  → 自己的會議（created_by = 自己）
  → OR 公開的（visibility = "public"）
  → OR 等級 3 的所有會議（上級往下看）
  → OR 同等級設為同等級可見的（visibility = "same_level" 且建立者為等級 2）

等級 3（一般使用者）：
  → 自己的會議（created_by = 自己）
  → OR 公開的（visibility = "public"）
  → OR 同等級設為同等級可見的（visibility = "same_level" 且建立者為等級 3）

既有資料（created_by = null）：
  → 視為公開，所有人可見
```

## 狀態轉換圖

### 角色變更

```
註冊（第一個帳號）→ 等級 1
註冊（後續帳號）  → 等級 3
                        │
                   等級 1 操作
                   ┌────┴────┐
                   ▼         ▼
              升為等級 2  降為等級 3
                   │         │
                   └────┬────┘
                        ▼
                   等級 1 操作
```

### 可見性變更

```
上傳時選擇（預設 private）
    │
    ├── public（公開）
    ├── private（僅自己）
    └── same_level（同等級可見）

上傳後可修改（上傳者本人 或 等級 1）
```

---

## 保護規則

- 等級 1 不能修改另一個等級 1 的角色
- 等級 2 不能操作等級 1 或其他等級 2 的帳號
- 至少保留一個等級 1 帳號
- 角色變更即時生效（get_current_user 從 DB 讀取最新 role）
