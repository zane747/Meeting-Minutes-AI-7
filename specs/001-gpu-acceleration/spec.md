# Feature Specification: GPU 加速模型推論

**Feature Branch**: `001-gpu-acceleration`
**Created**: 2026-03-25
**Status**: Draft
**Input**: User description: "目前這個專案貌似模型都是使用CPU再跑希望能讓GPU來分擔一點,目的是讓這個模型的效率更好"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 語音轉文字處理加速 (Priority: P1)

使用者上傳會議錄音進行本地語音轉文字處理時，系統自動利用可用的硬體加速資源，大幅縮短等待時間。使用者不需手動設定硬體，系統能自動偵測並使用最佳的運算資源。

**Why this priority**: 語音轉文字是最耗時的處理步驟，也是使用者最常等待的環節。加速此步驟能帶來最大的使用體驗提升。

**Independent Test**: 上傳一段 10 分鐘的會議錄音，觀察轉文字處理時間是否比純 CPU 模式顯著縮短，可獨立驗證加速效果。

**Acceptance Scenarios**:

1. **Given** 使用者擁有支援硬體加速的設備, **When** 上傳音檔進行本地轉文字, **Then** 系統自動使用硬體加速，處理速度比純 CPU 模式快至少 50%（參見 SC-001）
2. **Given** 使用者的設備不支援硬體加速, **When** 上傳音檔進行本地轉文字, **Then** 系統自動降級至 CPU 模式，功能正常運作且不顯示錯誤
3. **Given** 系統正在使用硬體加速處理音檔, **When** 加速資源記憶體不足, **Then** 系統優先嘗試載入較小模型繼續使用硬體加速；若仍不足才切換至 CPU 模式完成處理，不中斷使用者的工作流程

---

### User Story 2 - 說話者辨識處理加速 (Priority: P2)

使用者啟用說話者辨識功能時，系統利用硬體加速資源來加快辨識速度，減少多人會議的處理等待時間。

**Why this priority**: 說話者辨識是可選功能但計算密集。啟用後若太慢會嚴重影響使用體驗，加速此功能能讓更多使用者願意開啟。

**Independent Test**: 上傳一段多人對話的錄音並啟用說話者辨識，驗證辨識處理時間是否因硬體加速而縮短。

**Acceptance Scenarios**:

1. **Given** 說話者辨識功能已啟用且硬體加速可用, **When** 處理多人會議音檔, **Then** 辨識處理使用硬體加速，速度明顯提升
2. **Given** 硬體加速資源被語音轉文字佔用, **When** 同時需要進行說話者辨識, **Then** 系統透過排隊機制確保 Whisper 與 Diarization 不同時佔用 GPU，依序完成處理

---

### User Story 3 - 處理狀態與效能透明度 (Priority: P3)

使用者能透過系統狀態 API 端點查詢目前使用的運算資源（硬體加速或 CPU），以及處理效能資訊，讓使用者了解系統的運作狀態。

**Why this priority**: 透明度幫助使用者了解為何處理速度有差異，並幫助排除設定問題。

**Independent Test**: 啟動系統後呼叫系統狀態 API 端點，確認能看到目前使用的運算模式資訊。

**Acceptance Scenarios**:

1. **Given** 系統啟動完成, **When** 使用者查看系統狀態, **Then** 能看到目前使用的運算資源類型與相關資訊
2. **Given** 系統從硬體加速降級至 CPU, **When** 使用者查看系統狀態, **Then** 能看到降級原因與目前運作模式

---

### Edge Cases

- 硬體加速資源可用但驅動程式版本不相容時，GPU 偵測回傳不可用，系統自動使用 CPU 模式，並在啟動 log 中記錄警告
- 處理過程中硬體加速資源突然不可用（如被其他程式佔用）導致 OOM 時，適用與一般 OOM 相同的降級策略（FR-004），系統不區分 OOM 原因
- 同時有多個會議音檔排隊處理時，採嚴格排隊一次處理一個，其餘等待
- 硬體加速記憶體不足以載入較大的模型時，系統依降級順序（medium → small → base → tiny）嘗試較小模型繼續使用硬體加速，所有模型均 OOM 才退回 CPU（參見 FR-004）
- CPU fallback 後，下一個新請求從使用者設定的模型大小重新嘗試 GPU，降級僅影響當次請求
- PyTorch 未安裝時，系統視為無硬體加速環境，僅使用 CPU 模式

## Clarifications

### Session 2026-03-25

- Q: 記憶體不足時的模型處理策略？ → A: 自動降級至較小模型繼續使用硬體加速，仍不夠才退回 CPU
- Q: 多檔排隊處理策略？ → A: 嚴格排隊，一次只處理一個音檔，其餘等待
- Q: 運算狀態的顯示位置？ → A: 提供系統狀態頁面或 API 端點，使用者需主動查詢

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系統必須能自動偵測可用的硬體加速資源，並在啟動時記錄偵測結果。PyTorch 未安裝時視為無硬體加速環境，僅使用 CPU 模式
- **FR-002**: 系統必須在硬體加速可用時，自動將語音轉文字模型載入至加速裝置
- **FR-003**: 系統必須在硬體加速可用時，自動將說話者辨識模型載入至加速裝置
- **FR-004**: 系統必須在硬體加速記憶體不足（以捕捉 OOM 例外為準）時，依降級順序（預設 medium → small → base → tiny）嘗試較小模型繼續使用硬體加速；所有 GPU 模型均 OOM 才降級至 CPU 使用原始設定模型；若 CPU 也失敗則標記處理失敗。降級僅影響當次請求，下次請求從使用者設定的模型大小重新嘗試。降級過程對使用者靜默進行，降級紀錄可透過系統狀態 API（FR-007）事後查詢。降級日誌須包含：觸發時間、原始模型、降級後模型、降級原因、影響的處理任務 ID
- **FR-005**: 系統必須採用嚴格排隊機制，一次只處理一個音檔，確保 Whisper 與 Diarization 不同時佔用 GPU。Whisper 先執行，完成並釋放 GPU 後才執行 Diarization，兩者各自獨立處理 OOM
- **FR-006**: 系統必須提供環境設定選項，讓使用者可強制指定運算模式（自動偵測、強制硬體加速、強制 CPU）。使用者設定強制硬體加速但 GPU 不可用時，系統應在啟動時拋出明確錯誤訊息，告知安裝步驟
- **FR-007**: 系統必須提供系統狀態 API 端點，讓使用者可主動查詢目前使用的運算資源類型、VRAM 使用量、目前模型大小、降級紀錄等相關資訊
- **FR-008**: 系統必須確保安裝流程能正確引導使用者取得硬體加速所需的相依套件
- **FR-009**: 系統必須在每次模型使用完畢後釋放硬體加速資源，避免記憶體持續佔用。若釋放失敗，系統應記錄警告並繼續處理，不應導致服務中斷
- **FR-010**: 說話者辨識模型遇 OOM 時，直接退回 CPU 重新載入同一模型完成處理（pyannote 無多尺寸模型可降級），並記錄降級原因

### Key Entities

- **運算裝置 (Compute Device)**: 表示系統可使用的運算資源，包含類型（硬體加速/CPU）、記憶體容量、可用狀態
- **模型實例 (Model Instance)**: 表示載入的 AI 模型，包含模型大小、目前所在裝置、記憶體使用量
- **處理任務 (Processing Task)**: 表示一個音檔處理請求，包含所需模型、資源需求、目前狀態

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 10 分鐘會議音檔的語音轉文字處理時間，使用硬體加速後比純 CPU 模式減少至少 50%。基準條件：同一硬體環境、同一音檔、冷啟動（模型未預載）、使用相同模型大小，GPU 與 CPU 分別測量 3 次取平均值
- **SC-002**: 使用者從上傳音檔到取得完整會議紀錄的端到端時間，硬體加速模式下不超過音檔長度的 30%（例如 10 分鐘音檔在 3 分鐘內完成）
- **SC-003**: 偵測到 OOM 後，從捕捉例外到替代模型（較小 GPU 模型或 CPU 模型）成功載入並準備推論的間隔不超過 5 秒（不含推論本身的時間）
- **SC-004**: 使用者無需額外操作即可享受硬體加速帶來的速度提升（零設定體驗）
- **SC-005**: OOM 觸發降級時，系統不出現未捕捉的例外或程式崩潰，且最終產出處理結果（即使透過 CPU fallback）的成功率達 100%

## Assumptions

- 使用者的設備可能有也可能沒有支援硬體加速的運算資源，系統必須同時支援兩種情境
- 目標使用者的硬體加速記憶體約 6GB，需在此限制內選擇合適的模型大小
- 本地摘要生成服務（Ollama）獨立管理自身的硬體資源，不在 GPU 排隊機制範圍內，但在 Whisper/Diarization 推論前會主動卸載 Ollama 模型以釋放 VRAM
- 使用者主要在桌面環境使用本系統，非伺服器部署場景
- 安裝流程的改善僅需提供清晰的文件指引，不需實作自動安裝程式
- VRAM 臨界值情境由 OOM 例外機制自動處理，系統不進行預先 VRAM 檢查。若模型成功載入但推論中途 OOM，同樣觸發降級流程
- 模型降級會導致轉錄品質下降（tiny 遠低於 medium），系統優先確保處理完成而非品質。使用者可透過系統狀態 API 查詢實際使用的模型大小，評估是否需要重新處理
- 單機桌面場景下排隊數量有限，暫不設排隊超時機制
