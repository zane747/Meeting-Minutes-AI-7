# Feature Specification: 帳號管理

**Feature Branch**: `007-account-management`
**Created**: 2026-03-30
**Status**: Draft
**Input**: 主管課題 #2 — 帳號管理（CRUD：列表、檢視、修改密碼、停用/啟用、刪除）

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 檢視帳號列表 (Priority: P1)

已登入的使用者可以查看系統中所有帳號的列表，包含帳號名稱、狀態、建立時間等資訊。

**Why this priority**: 帳號列表是帳號管理的入口頁面，其他操作（修改、停用、刪除）都從這裡進入。

**Independent Test**: 登入後訪問帳號管理頁面，確認能看到所有帳號的列表。

**Acceptance Scenarios**:

1. **Given** 使用者已登入, **When** 訪問帳號管理頁面, **Then** 系統顯示所有帳號的列表（帳號名稱、狀態、建立時間）
2. **Given** 系統中有多個帳號, **When** 查看列表, **Then** 帳號依建立時間倒序排列（最新的在最上面）
3. **Given** 列表中有帳號, **When** 查看某個帳號, **Then** 可以看到該帳號的詳細資訊

---

### User Story 2 - 修改自己的密碼 (Priority: P1)

已登入的使用者可以修改自己的密碼。需要先輸入舊密碼驗證身份，再輸入新密碼。

**Why this priority**: 密碼修改是帳號管理最基本且最常用的功能，也是安全性的基礎。

**Independent Test**: 登入後進入修改密碼頁面，輸入舊密碼和新密碼，確認修改成功後可以用新密碼登入。

**Acceptance Scenarios**:

1. **Given** 使用者已登入, **When** 輸入正確的舊密碼和符合規則的新密碼, **Then** 密碼修改成功，顯示成功訊息
2. **Given** 使用者輸入錯誤的舊密碼, **When** 送出修改表單, **Then** 系統顯示「舊密碼不正確」錯誤訊息
3. **Given** 使用者輸入的新密碼少於 8 字元, **When** 送出修改表單, **Then** 系統顯示密碼長度不足的錯誤訊息
4. **Given** 密碼修改成功, **When** 使用者登出後重新登入, **Then** 必須使用新密碼才能登入，舊密碼無效

---

### User Story 3 - 停用與啟用帳號 (Priority: P2)

已登入的使用者可以停用其他帳號（使該帳號無法登入），也可以重新啟用已停用的帳號。

**Why this priority**: 停用帳號比刪除更安全——可以暫時禁止某人登入，但不會丟失資料，需要時可以恢復。

**Independent Test**: 停用某個帳號後，該帳號嘗試登入應失敗；重新啟用後應能正常登入。

**Acceptance Scenarios**:

1. **Given** 帳號 A 狀態為啟用, **When** 使用者點擊「停用」按鈕, **Then** 帳號 A 狀態變為停用，該帳號無法登入
2. **Given** 帳號 A 狀態為停用, **When** 使用者點擊「啟用」按鈕, **Then** 帳號 A 狀態恢復為啟用，可以正常登入
3. **Given** 使用者嘗試停用自己的帳號, **When** 點擊停用, **Then** 系統拒絕操作並顯示「無法停用自己的帳號」
4. **Given** 被停用的帳號目前已登入, **When** 該帳號下次發送請求, **Then** 系統強制登出該帳號

---

### User Story 4 - 刪除帳號 (Priority: P3)

已登入的使用者可以刪除其他帳號。刪除前需要二次確認。

**Why this priority**: 刪除是不可逆的操作，優先級最低。大多數情況下停用就夠了。

**Independent Test**: 刪除某個帳號後，該帳號從列表消失，且無法用該帳號登入。

**Acceptance Scenarios**:

1. **Given** 使用者要刪除帳號 A, **When** 點擊「刪除」按鈕, **Then** 系統彈出二次確認對話框
2. **Given** 使用者確認刪除, **When** 確認後, **Then** 帳號 A 從資料庫中移除，列表不再顯示
3. **Given** 使用者嘗試刪除自己的帳號, **When** 點擊刪除, **Then** 系統拒絕操作並顯示「無法刪除自己的帳號」

---

### Edge Cases

- 使用者修改密碼時，新密碼與舊密碼相同，系統如何處理？（允許，不強制不同）
- 停用帳號時，該帳號正在進行 AI 處理中的會議，如何處理？（讓處理完成，但阻止後續操作）
- 系統中只剩一個帳號時，能否停用或刪除？（禁止，確保系統至少有一個可登入的帳號）
- 刪除帳號時，該帳號上傳的會議紀錄如何處理？（v1 會議不綁定使用者，所以不受影響）

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系統必須提供帳號管理頁面，顯示所有帳號的列表（帳號名稱、狀態、建立時間）
- **FR-002**: 帳號列表必須依建立時間倒序排列
- **FR-003**: 系統必須提供修改密碼功能，需先驗證舊密碼正確才能修改
- **FR-004**: 新密碼必須符合與註冊時相同的規則（至少 8 字元）
- **FR-005**: 系統必須提供停用帳號功能，被停用的帳號無法登入
- **FR-006**: 系統必須提供啟用帳號功能，恢復被停用帳號的登入能力
- **FR-007**: 系統必須提供刪除帳號功能，刪除前需二次確認
- **FR-008**: 使用者不能停用或刪除自己的帳號
- **FR-009**: 系統必須確保至少有一個啟用狀態的帳號存在（防止所有帳號都被停用/刪除）
- **FR-010**: 帳號管理頁面必須受登入保護，未登入無法訪問

### Key Entities

- **User（使用者）**: 沿用 006-user-auth 建立的 User 資料表，利用 `is_active` 欄位控制帳號啟用/停用狀態

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 使用者可以在 30 秒內找到並查看任意帳號的資訊
- **SC-002**: 密碼修改流程可在 1 分鐘內完成
- **SC-003**: 停用帳號後，該帳號 100% 無法登入
- **SC-004**: 刪除操作 100% 需要二次確認，不會意外刪除

## Clarifications

### Session 2026-03-30

- Q: 修改密碼後，其他瀏覽器的 session 是否需要強制失效？ → A: 不需要，其他 session 維持有效直到自然過期（24 小時）

## Assumptions

- v1 所有使用者權限相同，任何登入的使用者都能管理所有帳號（權限管理是課題 #3 的範圍）
- 沿用 006-user-auth 的 User 資料表，不需要新增資料表
- `is_active` 欄位已存在於 User model 中（006-user-auth 已預留）
- 帳號管理頁面的入口放在導覽列
- v1 會議紀錄不綁定使用者，刪除帳號不影響會議資料
- 密碼修改後不強制重新登入，當前 session 和其他瀏覽器的 session 均維持有效直到自然過期（24 小時）
